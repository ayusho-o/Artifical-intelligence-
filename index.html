Prisma·Express·Next
Single-prompt generation was explicitly rejected by the spec. Multi-stage matters because each stage has a narrow responsibility and a strict output contract — intent extraction can't hallucinate DB columns, schema generation already knows the full entity graph. Errors are local and repairable without restarting the whole pipeline.
Repair Strategy
Instead of blind full retry: (1) detect the specific failure type, (2) target only the broken layer, (3) use a tighter JSON-only prompt with fewer output options, (4) merge the repaired part into existing schema. This keeps added cost to ~1 API call vs regenerating all 5 stages.
Determinism Techniques
JSON-only system prompts with explicit schema contracts. Per-stage output types. JSON fence + brace regex extraction with two-pass fallback. Explicit field lists prevent hallucination drift. Ambiguity resolved in Stage 1 before downstream stages see the data.
Failure Handling
Vague prompts: Stage 1 infers app_type and documents in assumptions[]. Conflicting requirements: documented in ambiguity_flags[], resolved with industry-standard default. Underspecified: missing_info[] captured, gaps filled with sensible defaults. Every decision is traceable in the schema output.
Execution Awareness
Stage 5 generates 5 real files: Prisma schema (ORM, directly runnable), Express router (drop-in Node.js routes), Next.js page stubs (mountable React pages), OpenAPI YAML (documentation + testing), .env.example (all required env vars). These are not simulated — they are syntactically valid code.
